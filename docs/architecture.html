Nemo Webex Bot

Bot conversationnel Webex connecte a un LLM cloud et a des outils MCP locaux — Mars 2026

1. Vue d'ensemble

Un bot Webex (nemo-drancy@webex.bot) heberge sur une VM interne recoit les messages des utilisateurs via websocket (pas de webhook, pas d'IP publique necessaire), les envoie a un LLM cloud (OVH AI Endpoints) pour le raisonnement, et donne au LLM acces a des outils locaux via le protocole MCP (Model Context Protocol).

Le LLM peut ainsi repondre a des questions factuelles (meteo, heure, droit francais, recherche web, production solaire) en appelant les outils MCP de facon autonome.

Pourquoi WebSocket et pas Webhook ? Les VMs internes n'ont pas d'IP publique. Le mode websocket de la lib webex-bot etablit une connexion sortante persistante vers le cloud Webex, ce qui ne necessite aucun port entrant ni reverse proxy.

3. Composants detailles

3.1 Webex Bot (webex-bot)

La bibliotheque Python webex-bot (v1.2.7, PyPI) gere la connexion websocket au cloud Webex, la reception des messages, et le dispatch vers des "commandes". On utilise un catch-all : une commande sans mot-cle enregistree comme help_command qui capture tous les messages.

Voir bot.py — Point d'entree du bot, classe LLMCommand, gestion de l'historique par utilisateur, boucle asyncio background.
Pattern threading important. La lib webex-bot a sa propre boucle asyncio interne. On ne peut pas utiliser asyncio.run() depuis ses callbacks. Solution : creer une boucle asyncio dediee dans un thread daemon, et utiliser asyncio.run_coroutine_threadsafe() pour executer les appels MCP/LLM depuis les callbacks synchrones de webex-bot.

3.2 Configuration

Voir config.py — Tokens, URLs des serveurs MCP, modele LLM, system prompt, constantes.
Voir bot.env — Variables d'environnement (secrets : WEBEX_BOT_TOKEN, OVH_TOKEN, TAVILY_API_KEY).

3.3 Client MCP (Model Context Protocol)

MCP est un protocole standardise (Anthropic, 2024) qui permet a un LLM d'appeler des "outils" (fonctions) exposes par des serveurs externes. Le transport utilise ici SSE (Server-Sent Events) : le client maintient une connexion HTTP persistante pour recevoir les messages du serveur, et envoie ses requetes via POST.

Voir mcp_client.py — Classe MCPManager, connexion SSE, decouverte des outils, appel d'outils, format OpenAI.
Pas de namespace. Contrairement a d'autres implementations, les noms d'outils ne sont pas prefixes par le serveur. Le MCPManager maintient un dictionnaire interne tool_name → server_name pour router les appels. C'est plus simple car les noms d'outils des differents serveurs ne se chevauchent pas.

3.4 Client LLM (OVH AI Endpoints)

Le LLM est appele via une API OpenAI-compatible. La boucle de function calling permet au LLM d'appeler plusieurs outils de facon iterative (max 5 iterations) avant de produire sa reponse finale.

Voir llm_client.py — Fonction call_llm(), boucle function calling, gestion des tool_calls et resultats.
Flux function calling (boucle LLM + outils) User: "Quel temps fait-il a Paris ?" | v LLM recoit le message + liste des outils disponibles | v LLM repond: tool_call → get_weather_today({"city_name": "Paris"}) | v Bot execute l'outil MCP → resultat meteo JSON | v LLM recoit le resultat de l'outil | v LLM genere la reponse finale: "A Paris il fait 12C, ciel couvert, vent 15 km/h..." | v Bot envoie la reponse dans Webex

4. Serveurs MCP (cote serveur)

Les 5 serveurs MCP tournent sur la VM 10.1.0.61 sous forme de services systemd. Chacun est un script Python utilisant FastMCP (la lib officielle Anthropic) qui expose des outils via le transport SSE.

Port Serveur Outils exposes Source de donnees
7001 horloge get_current_datetime Horloge systeme
7002 photovoltaic get_realtime, get_history... API Huawei FusionSolar
7003 legifrance search, get_article... API Legifrance (DILA/PISTE)
7004 weather get_current, get_forecast API Apple WeatherKit
7005 websearch web_search API Tavily (recherche web)

4.1 Structure d'un serveur MCP (exemple horloge)

from mcp.server.fastmcp import FastMCP
from datetime import datetime

mcp = FastMCP("horloge")

@mcp.tool()
def get_current_datetime() -> str:
    """Retourne la date et l'heure actuelles."""
    now = datetime.now()
    return now.strftime("Il est %H:%M le %A %d %B %Y")

if __name__ == "__main__":
    # Fix DNS rebinding pour acces depuis une autre machine
    from mcp.server.transport_security import TransportSecuritySettings
    mcp.settings.transport_security = TransportSecuritySettings(
        enable_dns_rebinding_protection=False
    )

    mcp.settings.host = "0.0.0.0"
    mcp.settings.port = 7001
    mcp.run(transport="sse")

4.2 Fix DNS Rebinding (important)

Probleme 421 Misdirected Request. Par defaut, FastMCP active la protection DNS rebinding quand host="127.0.0.1". Meme si on change ensuite host="0.0.0.0", la whitelist reste ["127.0.0.1:*", "localhost:*"]. Les requetes depuis une autre IP (le bot sur 10.1.0.55) sont rejetees avec HTTP 421.

La solution est d'ajouter dans le bloc if __name__ == "__main__" de chaque serveur :

from mcp.server.transport_security import TransportSecuritySettings
mcp.settings.transport_security = TransportSecuritySettings(
    enable_dns_rebinding_protection=False
)

Ce fix doit etre applique sur les 5 serveurs. Cote client (bot), on ajoute aussi le header Host correspondant a l'URL du serveur dans les requetes SSE, par precaution.

6. Creation du bot Webex

Avant de coder, il faut creer le bot sur le portail Webex :

  1. Aller sur developer.webex.com → My Webex Apps → Create a New App → Bot
  2. Remplir : nom, username (ex: nemo-drancy), icone, description
  3. Copier le Bot Access Token genere (il ne sera plus visible apres)
  4. Mettre ce token dans bot.env comme WEBEX_BOT_TOKEN
Le bot repond uniquement aux messages directs (DM) et aux mentions dans les espaces. En mode websocket, il n'a pas besoin de webhook URL ni d'IP publique.

7. Obtenir un endpoint LLM (OVH AI Endpoints)

  1. Creer un compte OVH et activer AI Endpoints dans la console
  2. Deployer un modele (ici gpt-oss-120b) — endpoint serverless
  3. Recuperer l'URL de l'endpoint et le token API
  4. L'API est compatible OpenAI : memes formats de requetes/reponses, meme function calling

Tout LLM avec une API OpenAI-compatible et le support du function calling peut etre utilise. Alternatives : OpenAI directement, Mistral AI (La Plateforme), ou un modele local via Ollama + litellm.

8. Points techniques

8.1 Modele de threading

Le bot utilise deux boucles asyncio separees :

BoucleThreadRole
Boucle webex-bot Thread principal Connexion websocket Webex, dispatch des messages
Boucle background (_loop) Thread daemon Appels MCP (SSE) et LLM (HTTPS) — tout l'async I/O

Les callbacks de webex-bot sont synchrones. Pour appeler du code async (MCP, LLM), on utilise la fonction helper run_async(coro) qui soumet la coroutine via asyncio.run_coroutine_threadsafe(coro, _loop) et attend le resultat. A noter : avant de creer le WebexBot, on appelle asyncio.set_event_loop(asyncio.new_event_loop()) pour eviter les conflits entre les deux boucles.

8.2 Historique par utilisateur

Chaque utilisateur Webex a son propre historique de conversation (via defaultdict(list) indexe par l'actor ID Webex). L'historique est limite a MAX_HISTORY (20) messages pour eviter de depasser la fenetre de contexte du LLM. C'est un stockage en memoire — l'historique est perdu au redemarrage du service. La date du jour est injectee dynamiquement dans le system prompt a chaque appel LLM.

8.3 Gestion des erreurs

Si un outil MCP echoue, l'erreur est renvoyee au LLM comme resultat de l'outil. Le LLM peut alors soit reformuler sa reponse sans l'outil, soit essayer un autre outil. Si le LLM lui-meme est injoignable, une exception remonte et webex-bot renvoie un message d'erreur generique.

9. Pour reproduire ce setup

Checklist pour deployer un bot similaire dans un autre contexte :

  1. Creer le bot Webex sur developer.webex.com, recuperer le token
  2. Obtenir un endpoint LLM avec support function calling (OVH, OpenAI, Mistral...)
  3. Deployer les serveurs MCP (FastMCP + SSE) sur une machine accessible depuis le bot. Desactiver la DNS rebinding protection si machines differentes.
  4. Installer les dependances : pip3 install webex-bot httpx mcp python-dotenv
  5. Adapter config.py : URLs des serveurs MCP, endpoint LLM, system prompt
  6. Creer bot.env avec les tokens
  7. Tester manuellement : python3 bot.py puis envoyer un DM au bot dans Webex
  8. Creer le service systemd pour le demarrage automatique

10. Creer ses propres outils MCP

Pour ajouter un outil au bot, il suffit de creer un nouveau serveur MCP (ou d'ajouter un outil a un serveur existant) :

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("mon-serveur")

@mcp.tool()
def ma_fonction(param1: str, param2: int = 10) -> str:
    """Description visible par le LLM — soyez precis."""
    # Logique metier ici
    return f"Resultat pour {param1} avec valeur {param2}"

if __name__ == "__main__":
    from mcp.server.transport_security import TransportSecuritySettings
    mcp.settings.transport_security = TransportSecuritySettings(
        enable_dns_rebinding_protection=False
    )
    mcp.settings.host = "0.0.0.0"
    mcp.settings.port = 7006  # nouveau port
    mcp.run(transport="sse")

Puis ajouter l'URL dans config.py du bot :

MCP_SERVERS = {
    ...
    "mon-serveur": "http://10.1.0.61:7006/sse",
}

Redemarrer le bot — le nouvel outil est automatiquement decouvert et propose au LLM.
